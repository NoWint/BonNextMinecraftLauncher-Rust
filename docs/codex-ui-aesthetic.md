# CodeX UI 设计审美深度总结

> 基于 CodeX 桌面应用（macOS 客户端）主界面与设置页两张截图的视觉与交互拆解。
> 目的：为 BonNext 的 UI 演进提供可复用的审美与组件参考。

---

## 一、整体定位

CodeX 把"开发者工具"做成了"专业级创作软件"的质感——不是 VS Code 那种堆满图标的工程化界面，也不是 ChatGPT 网页那种对话卡片，而是一种**克制、桌面优先、长时间停留不疲劳**的视觉语言。

关键词：

- **Sub-pro（克制专业）**：拒绝炫技、拒绝渐变光晕
- **Editor-first（编辑器优先）**：内容区占绝对主导，控件为内容服务
- **Quiet chrome（安静外壳）**：导航、状态、装饰一律退到背景层
- **One-glance comprehension（一瞥即懂）**：任何状态下信息密度可被快速扫读

## 二、色彩系统

### 2.1 主色板（截图取色）

| 角色          | 取值（估）   | 用途                               |
| ------------- | ------------ | ---------------------------------- |
| 画布底色      | `#0F0F10`    | 主内容区背景                       |
| 侧栏底色      | `#17171A`    | 略亮一档，让导航与内容分层         |
| 分隔线        | `#26262A`    | 1px 细分隔                         |
| 卡片底        | `#1C1C20`    | 命令块、设置项卡片                 |
| 卡片悬停/选中 | `#24242A`    | 列表行 hover                       |
| 文字主        | `#ECECEC`    | 标题与正文                         |
| 文字次        | `#9A9AA2`    | 描述、辅助信息                     |
| 文字辅        | `#6B6B72`    | placeholder、极弱说明              |
| 强调黄        | `#F5C518`    | 状态点、极高级别徽标               |
| 强调蓝        | `#3B82F6`    | 开关 ON、链接、激活状态            |
| 危险/红字     | `#F25C5C`    | JSON 字符串、错误                  |
| 代码绿        | `#A6E3A1`    | JSON 数字（局部可见）              |
| 背景光晕      | 极弱橙紫渐变 | 仅在窗口最外缘作环境光，不喧宾夺主 |

### 2.2 用色原则

1. **不黑到底**：避免纯 `#000`，选用 `#0F0F10` 之类的"近黑炭灰"，使发光描边、玻璃面板仍有边界。
2. **中性灰主导**：90% 像素是中性的炭灰/白/灰；任何彩色都带"任务含义"。
3. **单点强调**：整屏只允许 1 个高饱和色块（开关、状态点、当前选中），其余保持灰阶。
4. **代码高亮收敛**：截图里 JSON 高亮只有 2-3 种颜色（红字符串、白 key、灰标点），不追求 IDE 那种彩虹色。
5. **环境光代替纯色背景**：窗口边缘隐约有橙红 → 紫蓝的辉光，暗示"被环境照亮"而非"被涂了色"。

## 三、字体系统

### 3.1 字体族

- **英文/数字**：`SF Pro Text` / `SF Pro Display`（系统字体）
- **中文**：`PingFang SC`
- **代码/数据**：`SF Mono` / `JetBrains Mono`（命令行块、JSON 文件视图）
- **绝不混用衬线**：所有 UI 元素统一无衬线

### 3.2 字号阶梯（em，base=16px）

| 用途       | px  | 备注             |
| ---------- | --- | ---------------- |
| 页面大标题 | 22  | 工作树/常规      |
| 区块标题   | 15  | 工作树根目录     |
| 正文       | 13  | 描述、说明       |
| 列表项主文 | 13  | 侧栏、设置项     |
| 次要信息   | 12  | 项目副标题       |
| 标签/徽标  | 11  | 完全访问、极高   |
| 代码       | 12  | 命令块、文件视图 |
| 极小提示   | 11  | 空状态提示       |

### 3.3 字重与排版细节

- **常规 400** 占绝大多数；**中粗 500** 用于"当前选中/强调标题"，**不出现 700+**。
- 中文标题可考虑用 500 替代加粗，避免笔画糊在一起。
- 数字常使用 **tabular-nums**（等宽数字），让行号、参数对齐。
- 行高约 1.45–1.55，段间距用 12/16/24 三档节制。
- 颜色用"灰度阶梯"区分层级，而非字号突跃——13px 主 + 13px 灰、12px 灰、11px 极灰。

## 四、布局与栅格

### 4.1 三栏式（主界面）

```
┌──────────┬────────────────────────────┬──────────────┐
│ Sidebar  │  Editor / Composer Area    │  Inspector   │
│ 240px    │  flex: 1                   │  360-420px   │
│          │                            │              │
└──────────┴────────────────────────────┴──────────────┘
```

- **侧栏**：固定宽，可折叠；只有图标 + 单行文字；项目/任务分组用更弱的灰字标识。
- **中栏**：唯一的内容承重墙；上下两端固定（顶栏 + 输入条），中间滚动。
- **右栏**：信息附加位（文件、上下文、属性），**不抢戏**——没有也行，有则增益。

### 4.2 双栏式（设置页）

```
┌──────────┬────────────────────────────┐
│ Sidebar  │  Settings Form Area        │
│ 240px    │  居左对齐，最大宽 ~720px   │
│          │                            │
└──────────┴────────────────────────────┘
```

- 设置项不铺满全宽，而是**约 720px 居中**，左侧大块留白，制造"翻书"节奏。
- 卡片之间用 16-20px 纵向间距，卡片内部 16-20px 内边距。

### 4.3 关键栅格参数

- **基础间距单位**：4px（4 / 8 / 12 / 16 / 20 / 24 / 32 / 48）
- **侧栏项高度**：32-36px
- **设置项高度**：约 56-64px（含主标题 + 描述两行）
- **输入框/按钮高度**：28-32px（比通用 40px 偏小一档，**显紧凑**）
- **圆角**：6px（标签/小按钮）/ 8px（输入框、卡片）/ 12px（命令块等大卡片）
- **分隔线**：1px `#26262A`；避免用阴影分层

### 4.4 窗口装饰

- 顶部 macOS 红黄绿按钮保留——尊重宿主系统
- 自定义 chrome 区域顶栏高 36-40px，**不画线**，仅靠与画布同色的细微差异
- 顶栏左侧是上下文面包屑（命令路径 / 文件路径），右侧是辅助操作

## 五、组件语言

### 5.1 导航（Sidebar）

- **Logo + 全局搜索**：顶部固定，"Codex" wordmark + 放大镜图标
- **主操作区**：新建任务 / 已安排 / 插件 / 拉取请求——图标 + 单行，激活态有左侧 2px 蓝/黄竖条
- **分组标题**：纯灰 12px，**无图标、无背景**，仅靠留白分组
- **二级项**：项目名、任务名——缩进一级，hover 出 8% 白底
- **底部固定区**：设置 + 信息（关于）

### 5.2 命令/对话块（Editor Card）

- 浅卡片底 `#1C1C20`，圆角 12px
- 内部代码用 mono 字体 + 灰白文字
- 右上角小图标按钮（复制、编辑），hover 时变亮
- **不带左侧色条、不带时间戳**——和对话气泡完全区分

### 5.3 输入条（Composer）

- 固定底栏，背景与画布同色或更深一档 `#0A0A0B`
- 多行输入，**占位提示在左上**（"要求后续变更"）
- 底部一行操作：左 `+` 加号 → 中间 `完全访问` 权限徽标 → 右模型/模式选择 + 圆形发送按钮
- 圆形发送按钮 28-32px，主色蓝或黄，**带轻微内阴影/外发光**

### 5.4 设置项（Form Row）

```
┌───────────────────────────────────────────────┐
│ 工作树根目录                       [ 默认  ▾] │
│ ChatGPT 创建代理工作树的目录；留空使用默认位置  │
└───────────────────────────────────────────────┘
```

- 标题 15px 中粗 / 描述 12px 灰 / 控件右对齐
- 整行高度 56-64px，**点击行任意位置**都能触发控件
- 同一卡片里多行设置用 1px 细线分隔，不嵌套卡片

### 5.5 开关（Toggle）

- iOS 风格，轨道 36×20，圆形 thumb 16px
- ON：蓝 `#3B82F6`；OFF：`#3A3A40`
- **开关后没有任何状态文字**（不像某些设计写"开/关"），状态由颜色传达

### 5.6 选择器（Select / Dropdown）

- 极简：单行文字 + `▾` 箭头，背景透明或与卡片同色
- hover 出现 `#24242A` 底
- 数字输入直接用 inline `<input type="number">` 风格边框，无 spinner 按钮

### 5.7 空状态

- 一行灰字 + 刷新图标
- 不放插画、不放 emoji
- 例："尚无工作树" + 右上角刷新按钮
- 例："ChatGPT 创建的工作树将显示在此处"——直接陈述事实，不情绪化

### 5.8 徽标/标签

- "5.6 Sol 极高" 这类状态：色点 + 文字
- 完全访问 / 插件状态：用单色描边胶囊，高度 20-22px
- 不用纯色填充背景——保持"标签很轻"的感觉

## 六、交互与动效

### 6.1 反馈原则

- **hover**：背景 8% 提亮，无边框、无阴影变化
- **active/press**：背景再压暗 4%，模拟按下
- **focus**：1.5px 蓝色外描边，**禁止默认浏览器 outline**
- **disabled**：透明度 0.4，禁 hover

### 6.2 过渡

- 时长 120-180ms，曲线 `cubic-bezier(0.16, 1, 0.3, 1)`（Swift 标准 ease-out）
- 折叠侧栏、卡片展开等场景用 height/opacity 同步过渡
- **不滥用 spring**——保持线性或单一缓出，避免"玩具感"

### 6.3 命令面板/搜索

- 顶部居中弹出，宽 560-640px，圆角 12px，背景 `#17171A` + `backdrop-filter: blur(20px)`
- 列表项 36-40px 高，左侧图标 + 主文 + 右侧快捷键
- 选中项左侧 2px 蓝色条 + 背景提亮 6%

## 七、信息架构

### 7.1 主界面

```
左侧：任务流（新建/已安排/插件/拉取请求）       ← 操作维度
中栏：当前任务的对话流                          ← 创作维度
右栏：当前选中对象的属性/文件                   ← 上下文维度
```

三栏从"做什么"→"在做"→"做的是什么"形成完整闭环，**永远不出现"我该去哪"的问题**。

### 7.2 设置页

```
个人 (Personal)      → 与用户相关的偏好
  ├ 常规              → 启动、默认行为
  ├ 外观              → 主题、密度
  ├ 语音              → TTS/STT
  ├ 配置              → 模型/订阅
  ├ 个性化            → 个性化推荐
  ├ 宠物              → 彩蛋
  ├ 键盘快捷键
  └ 账户
集成 (Integrations)  → 第三方连接
  ├ 应用快照
  ├ 插件
  ├ 浏览器
  └ 电脑操控
编码 (Coding)        → 开发者向
  ├ 钩子
  ├ 连接
  ├ Git
  ├ 环境
  └ 工作树
```

- **三级标题用更小更灰的字 + 大量留白分隔**（无背景框、无分割线），类似 Apple 的"系统设置"分组方式
- 顶部一个全局 `← 返回应用` 按钮 + 搜索框，**永远可达**

## 八、视觉细节备忘录

1. **极弱的环境光**：窗口边缘隐约有暖色辉光——但只占像素 <3%，绝不主导。
2. **无插画、无 emoji、无品牌色大块面**：整个界面接近"软件质感"而非"营销质感"。
3. **代码用 mono，UI 用 sans**：泾渭分明，**绝不**把 mono 用在按钮或标签上。
4. **数字常带单位/状态字**：`5.6 Sol 极高` 而非 `5.6`；`15` 工作树数 而非数字孤悬。
5. **行号、文件路径、面包屑**一律 mono 字体并灰化——**结构信息降噪**。
6. **图标尺寸 16/20/24px 三档**，线宽统一 1.5px，圆角端点（Lucide 风格）。
7. **避免阴影**：分层靠背景色差 + 1px 边线，**绝不用 `box-shadow`** 制造浮起感。
8. **玻璃面板仅在命令面板/浮层**用 `backdrop-filter`，主画布永远纯色，**避免大面积模糊带来的性能/视觉噪点**。
9. **按钮无填充**：强调按钮用主色填充（发送、激活），次要按钮仅文字 + hover 底色。
10. **状态用色点不用图标**：`极高` 是一个小色点 + 文字，不画一个温度计或火焰。

## 九、可以直接借鉴的 12 条规则

1. **底色用近黑炭灰 `#0F0F10`，不用纯黑**。
2. **侧栏比画布亮 4-6%，靠背景色分层，不用分隔线**。
3. **字号梯度小（11/12/13/15/22），靠灰度区分层级**。
4. **强调只用一种颜色，整屏只允许 1 个高饱和色块**。
5. **代码用 mono，UI 用 sans，泾渭分明**。
6. **数字用等宽字体 + tabular-nums**。
7. **行高 1.45-1.55，段间距用 12/16/24 三档**。
8. **圆角 6/8/12 三档，不混用**。
9. **hover 仅 8% 提亮；focus 用 1.5px 蓝外描边**。
10. **过渡 120-180ms，缓出曲线 `cubic-bezier(0.16, 1, 0.3, 1)`**。
11. **不滥用阴影，必要时用 1px 边线**。
12. **空状态一句话陈述事实，不情绪化，不放插画**。

## 十、BonNext 中的潜在落地点

> 以下为基于现有 BonNext 的现状（ZZZ 赛博朋克 + 黄黑 + 切角）如何借鉴 CodeX 审美的建议。

| BonNext 模块     | 借鉴点                                                | 注意                       |
| ---------------- | ----------------------------------------------------- | -------------------------- |
| `AppShell` 侧栏  | 紧凑 32-36px 项高、单行文字 + 图标、激活态左侧 2px 条 | 保留切角装饰作为品牌差异点 |
| `SettingsPage`   | 三级分组小标题 + 大留白、行级设置项 56-64px           | 不抢现有黄色强调           |
| `DownloadPanel`  | 行级 hover 8% 提亮、不画阴影、状态用色点              | 保留 speed/eta 数据字体    |
| `InstanceDetail` | 右栏上下文 + 中栏内容 + 左栏操作的三栏复用            | 切角用于强调卡而非装饰卡   |
| 空状态           | 一行灰字 + 刷新图标，**撤掉所有插画**                 | 与"赛博朋克"折中           |
| 模态/命令面板    | 玻璃面板 `backdrop-filter: blur(20px)` + 圆角 12      | 黄黑主题下用 0.3 透明度    |

### 反向警告（不要照搬的点）

- **不要把底色从 #0E0E0E 改成 #0F0F10 那种炭灰再统一**——BonNext 是赛博朋克，更深、更黑、更电。
- **不要撤掉切角 clip-path**——那是品牌护城河。
- **不要用 SF Pro / PingFang 替 Bebas Neue 标题**——保留 Bebas Neue 做大标题，Body 仍可用 Inter。
- **不要在强调色上让步**——CodeX 的克制是因为它要承载文字；BonNext 的游戏启动器需要情绪化锚点，#FFE600 黄必须留。
